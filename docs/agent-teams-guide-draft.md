# Claude Code Agent Teams 활용 가이드 (초안)

## 개요

Claude Code Agent Teams는 단일 Claude Code 세션을 여러 에이전트로 분리하여 병렬/협업 개발을 수행하는 기능입니다.
2026년 2월 Opus 4.6과 함께 Research Preview로 공개되었습니다.

## 핵심 개념

### 구조
- **팀 리드**: 작업 조율 및 할당을 담당하는 메인 세션
- **팀메이트**: 각각 독립된 컨텍스트 윈도우에서 작업하는 에이전트
- 사용자는 리드를 거치지 않고 개별 팀메이트와 직접 상호작용 가능

### 활성화
```bash
# 환경 변수로 활성화
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

### 디스플레이 모드
- **auto** (기본): tmux 사용 중이면 split panes, 아니면 in-process
- **in-process**: 단일 터미널에서 Shift+Up/Down으로 팀메이트 전환
- **split panes**: tmux/iTerm2에서 각 팀메이트가 별도 패널

## 자율 개발에 적합한 작업 vs 부적합한 작업

### 적합한 작업
- CRUD API 엔드포인트 생성 등 패턴이 명확한 반복 작업
- 동일 구조의 클래스 N개 생성
- 테스트 코드 작성
- 파일/변수 일괄 리네이밍
- 원본이 있는 포팅 작업 — 단, 인터페이스 명세와 완료 기준이 충분히 갖춰진 경우에 한함

### 지시가 더 상세해야 하는 작업
- 도메인 지식이 필요한 구현 (게임 로직, 비즈니스 로직 등) — 자율 실행 자체는 가능하나, 지시의 구체성이 결과 품질을 좌우함
- 원본이 있는 포팅이라도 모듈 간 연결, 완료 기준이 불명확하면 실패함

### 부적합한 작업 (대화형이 더 적합)
- 설계 판단이 필요한 리팩토링
- 구조 결정이 포함된 신규 기능

### 판단 기준
> "검증 기준을 명확히 줄 수 있는가?"
> 빌드 성공, 테스트 통과 같은 객관적 기준이 있으면 자율 실행이 잘 동작하고,
> 사용자 판단이 필요하면 대화형이 적합하다.

## 에이전트 팀 설계 원칙

### 핵심 원칙: 에이전트는 명시적으로 적힌 것만 한다

에이전트에게 암묵적 기대는 통하지 않는다:
- "알아서 연결하겠지" → 연결하지 않음
- "원본을 보면 알겠지" → 분석 문서만 보고 대충 만듦
- "빌드 되면 되겠지" → 스텁으로 빌드 통과시킴

따라서:
1. 호출 관계, 데이터 흐름, 초기화 순서를 **빠짐없이** 적어야 함
2. 암묵적 기대가 아니라 명시적 검증 기준을 제시해야 함
3. "~하지 말 것" (스텁 금지, placeholder 금지) 도 명시해야 함

### 역할 분리
- 에이전트마다 **수정 범위**를 명확히 지정 (폴더/파일 단위)
- 에이전트 간 수정 범위가 겹치지 않도록 설계
- 각 에이전트가 담당할 원본 파일을 명시적으로 지정

### 모듈 간 인터페이스를 사전 정의할 것
수정 범위만 적으면 "누가 누구의 어떤 함수를 호출하는지"가 빠진다.
에이전트 간 호출 관계를 CLAUDE.md에 명시해야 한다.
```
예시:
- Agent 3은 ResourceManager::DoSound(ESoundId)를 호출하여 사운드를 재생한다.
- Agent 4는 ObjectManager::mCharacters를 순회하여 렌더링한다.
- Agent 2는 Application.cpp에서 모든 모듈의 Initialize()를 호출 순서대로 연결한다.
```

### 통합 단계를 반드시 포함할 것
각 에이전트가 자기 폴더만 잘 만들어도, 아무도 연결하지 않으면 빌드만 되는 껍데기가 된다.
작업 단계에 모듈 통합 단계를 명시적으로 포함해야 한다.
```
예시:
1단계: 분석 + 뼈대 (병렬)
2단계: 게임로직 + 렌더링 + 부가기능 (병렬)
3단계: ★ 모듈 통합 (뼈대 에이전트가 메인에서 전체 연결)
4단계: 코드 리뷰 + 포팅 검증
```

### 에이전트 분할 기준을 원본 코드 구조와 맞출 것
- 폴더 단위 분할이 깔끔해 보이지만, 원본 코드는 한 파일 안에 로직+렌더링이 섞여 있을 수 있다
- 한 원본 파일의 로직+렌더링을 한 에이전트가 모두 담당해야 누락이 줄어든다
- 단, 이 경우 에이전트 간 파일 충돌 위험이 있어 트레이드오프 판단 필요

### 영역이 걸치는 파일의 담당자를 명시할 것
- 원본 파일 안에 여러 에이전트 영역의 코드가 섞여 있는 경우, **담당 에이전트가 전부 포팅**하되 배치 위치를 명시해야 한다
- 예: mainmenu.pas의 렌더링 함수(DrawCharacter, ShowOption 등)는 게임 로직 에이전트가 포팅하되 `src/Render/`에 배치
- 이를 위해 해당 에이전트의 수정 범위에 예외를 추가한다 (예: "담당 원본 파일 내 렌더링 함수에 한해 src/Render/ 수정 가능")
- 명시하지 않으면 서로 상대방이 할 거라고 생각하고 누락된다

### 통합 단계에서는 수정 범위를 해제할 것
- 통합은 모든 폴더에 걸칠 수밖에 없다 (Application.cpp에서 각 모듈 연결, 필요 시 새 클래스 생성 등)
- 통합 담당 에이전트의 수정 범위 제한을 명시적으로 해제해야 한다
- 예: "3단계(통합)에서 Agent 2는 모든 src/ 하위 폴더를 수정할 수 있다"

### 의존성 관리
- 완전 병렬이 어려운 경우 단계를 나눠서 실행
- 예: 1단계(분석 + 뼈대) → 2단계(로직 + 렌더링 병렬)
- 뼈대/인프라가 먼저 완성되어야 나머지가 작업 가능

### 리뷰 에이전트
- 구현 에이전트와 별도로 리뷰 전담 에이전트를 두면 코드 품질 유지에 효과적
- 리뷰 관점이 다르면 분리하는 것이 좋음
  - 예: 코드 규칙 리뷰 vs 포팅 완전성 검증
- 한 에이전트가 두 관점을 동시에 보면 하나에 집중하다 다른 쪽을 놓칠 수 있음

### 에이전트 분할의 판단 기준
- **쪼개지 말아야 할 경우**: 한 에이전트가 다른 에이전트의 출력을 입력으로 받아야 하는 경우 (순환 의존성 발생)
- **쪼개도 되는 경우**: 서로 독립적으로 완성 가능한 단위 (예: 핵심 게임 로직 vs 설정/크레딧 등 부가 기능)
- 범위가 넓더라도 하나의 에이전트가 순차 처리하되, 리뷰 에이전트가 중간 체크하는 것이 더 깔끔할 수 있다

### 런타임 의존성 배치를 명시할 것
- 빌드에는 필요 없지만 실행에는 필요한 파일(리소스, 설정 파일, 데이터 파일 등)의 배치를 명시적으로 지시해야 한다
- "어떤 에이전트가, 어떤 파일을, 어디에 복사/가공하는지"를 CLAUDE.md에 적어야 한다
- 예: "Agent 4는 original/resources/의 리소스를 resources/에 복사하고, DOGS.PX를 파싱하여 사용 가능한 형태로 변환한다"
- 폴더 구조에 `resources/`를 정의해 놓고 누가 채우는지 적지 않으면, 아무도 하지 않는다

### 에이전트가 막혔을 때의 대응 방법
- 다른 에이전트의 미완성 코드 때문에 진행할 수 없는 경우, **스텁을 만들지 말고 팀 리드에게 보고**할 것
- 보고 내용: 어떤 에이전트의 어떤 인터페이스가 필요한데 아직 없는지를 명시
- 팀 리드가 해당 에이전트에게 우선 처리를 지시하거나, 작업 순서를 조정
- "스텁 금지"와 "막히면 보고" 원칙을 CLAUDE.md에 명시해야 한다

### 에이전트 간 결과 전달 방법
Agent Teams에서 에이전트 간 피드백 전달은 **팀 리드를 경유**하여 이루어진다:
- 리뷰/검증 에이전트가 문제를 발견하면, 팀 리드에게 보고
- 팀 리드가 해당 구현 에이전트에게 수정을 지시
- 파일 기반 전달이 필요한 경우, `docs/reviews/` 등 약속된 경로에 리뷰 결과를 마크다운으로 작성하고 해당 에이전트가 참조하도록 지시

## 완료 기준 설계

### 빌드 성공만으로는 부족하다
"빌드 성공"이 유일한 검증 기준이면, 에이전트는 스텁이든 placeholder든 빌드만 통과하면 "완료"로 처리한다.

### 완료 기준에 포함해야 할 것
1. **스텁/placeholder 금지**: 빈 구현, `return false;`, TODO 주석만 있는 함수는 완료로 인정하지 않는다
2. **원본 1:1 대응** (포팅의 경우): 각 함수는 원본의 해당 함수와 1:1 대응해야 한다. 분기, 상수값, 호출 순서가 동일해야 완료로 인정한다
3. **빌드 성공**: /W4 경고 0개, 에러 0개

### 에러 핸들링/로깅 기준을 포함할 것
- 빌드 성공만 기준이면, 런타임 실패를 조용히 무시하는 코드도 "완료"로 통과한다
- 리소스 로딩 실패, 파일 미존재 등 런타임 오류 시 원인을 확인할 수 있는 최소한의 로깅/알림을 요구해야 한다
- 예: "리소스 로딩 실패 시 OutputDebugString 또는 MessageBox로 파일명과 실패 원인을 표시할 것"
- 이 기준이 없으면 실행 시 검은 화면만 나와도 원인 파악이 불가능하다

### 검증 에이전트의 검증 깊이를 명시할 것
- "함수 존재 여부"가 아니라 **"함수 내용 일치 여부"**를 검증해야 한다
- 원본 함수의 각 분기/조건문이 C++ 코드에 존재하는지 줄 단위로 대조
- 상수값(좌표, 인덱스, 확률 등)이 원본과 일치하는지 확인

## CLAUDE.md 작성 가이드

Agent Teams에서 CLAUDE.md는 모든 에이전트가 참조하는 공통 지침서 역할을 합니다.

### CLAUDE.md는 살아있는 문서이다
CLAUDE.md는 한 번 작성하고 끝이 아니라, 단계별 점검 시마다 갱신하는 문서이다.
- 1단계 완료 후: 분석 결과를 바탕으로 모듈 간 호출 관계를 구체적인 클래스명::함수명으로 업데이트
- 2단계 완료 후: 스텁 발생 여부, 에이전트 분할 적절성 등을 반영하여 수정
- 단계가 진행될수록 CLAUDE.md는 더 구체적이 된다

### 포함해야 할 내용
1. **프로젝트 개요**: 무엇을 하는 프로젝트인지
2. **기술 스택**: 언어, 프레임워크, 빌드 도구, 플랫폼
3. **프로젝트 구조**: 폴더 구조와 각 폴더의 역할
4. **에이전트 팀 구성**: 각 에이전트의 역할, 대상 파일, 수정 범위, 의존성
5. **모듈 간 인터페이스**: 에이전트 간 호출 관계, 데이터 흐름, 초기화 순서
6. **작업 단계**: 어떤 순서로 에이전트가 실행되는지 (통합 단계 포함)
7. **완료 기준**: 빌드 성공 + 스텁 금지 + 원본 대응 등 구체적 기준
8. **원칙/제약**: 코드 규칙, 금지사항 등

### 에이전트 정의 시 필수 항목
- 역할 이름과 설명
- **대상 원본**: 담당할 원본 파일 목록
- **수정 범위**: 수정 가능한 폴더/파일 경로
- **의존성**: 어떤 에이전트의 작업이 완료되어야 시작 가능한지
- **다른 에이전트와의 호출 관계**: 어떤 에이전트의 어떤 인터페이스를 사용하는지

## 실전 사례: CyberDogs 포팅

### 프로젝트 특성
- DOS용 터보 파스칼 7.0 → Windows C++ / Direct2D 포팅
- 원본 소스 20개 파일, SPX 라이브러리 참조
- 자체 포맷 리소스 (DOGS.PX)

### 1차 시도 에이전트 구성 (7개)

| # | 역할 | 담당 파일 | 수정 범위 | 단계 |
|---|------|----------|----------|------|
| 1 | 분석 (Analyzer) | 전체 원본 소스 + SPX 라이브러리 | docs/ | 1단계 |
| 2 | 뼈대 (Scaffold) | dogs.pas, screen.pas, keyboard.pas, joystick.pas, stick.pas, globals.pas | src/Core/, src/Utils/ | 1단계 |
| 3 | 게임 로직 (GameLogic) | game.pas, gamearea.pas, elements.pas, equip.pas, mainmenu.pas | src/Game/ | 2단계 |
| 4 | 리소스/렌더링 (Renderer) | pics.pas, sounds.pas, buffer.pas, bigfont.pas, fancy.pas | src/Resource/, src/Render/ | 2단계 |
| 5 | 부가 기능 (Support) | config.pas, ini.pas, credits.pas, hallfame.pas | src/Game/ | 2단계 |
| 6 | 코드 리뷰 (CodeReviewer) | - | 리뷰만 수행 | 상시 |
| 7 | 포팅 검증 (PortingVerifier) | - | 리뷰만 수행 | 상시 |

### 1차 시도 결과
- 빌드는 성공했으나 **실제 게임 동작 재현에 실패**
- 포팅 완성도 약 92%(Agent 7 기준)이나, 모듈 간 연결 부재로 메인 메뉴조차 원본과 다르게 동작

### 1차 시도에서 발견된 문제점

**1. 모듈 간 통합 에이전트 부재**
- 각 에이전트의 수정 범위가 폴더 단위로 격리됨
- Application.cpp의 Update/Render 콜백이 플레이스홀더 상태로 남음
- 아무도 모듈을 연결하는 역할을 맡지 않았음

**2. 인터페이스 명세 부재**
- "수정 범위"는 있었지만, "누가 누구의 어떤 함수를 호출하는지"가 없었음
- 결과: 스텁 함수(`return false;`, 빈 구현)가 대량 발생

**3. 완료 기준이 "빌드 성공"에 머물렀음**
- 에이전트는 빌드만 통과하면 스텁이든 placeholder든 "완료"로 처리
- 원본 동작과의 일치 여부는 검증하지 않음

**4. 원본 참조 방식의 한계**
- 에이전트들이 원본을 줄 단위로 대조하지 않고, 분석 문서(docs/)를 기반으로 C++ 구조를 만듦
- 세부 상수값, 좌표, 분기 조건이 누락됨

**5. 검증 에이전트의 검증 깊이 부족**
- "함수가 존재하는지" 수준의 검증이었고, 함수 내부 로직의 줄 단위 일치는 확인하지 않음

**6. 원본 함수가 여러 에이전트 영역에 걸치는 경우**
- 예: mainmenu.pas의 한 함수에 로직(Agent 3 영역)과 렌더링(Agent 4 영역)이 섞여 있음
- 어느 쪽도 완전히 포팅하지 못함

### 핵심 교훈

> **에이전트 팀은 "무엇을 만들지"뿐 아니라 "어떻게 연결되는지"와 "무엇이 완료인지"까지 명시해야 멈추지 않고 완성한다.**

## 권장 실행 흐름

계획을 한 번에 세워서 전부 실행시키는 것이 아니라, 단계마다 검토 지점을 두어야 한다.
원본 코드를 실제로 포팅해보기 전에는 모듈 간 의존성을 완전히 파악하기 어렵기 때문이다.

### 전체 흐름

```
1. CLAUDE.md 작성 (계획 수립)
       ↓
2. 실행 세션(CLI에서 새로 열어 CLAUDE.md를 읽게 한 세션)에게 계획 검토 요청 ← "이 계획대로 실행하면 빠진 것이 있는지 검토해줘. 실행하지 마."
       ↓
3. 피드백 반영하여 CLAUDE.md 수정
       ↓
4. 1단계 실행 (분석 + 뼈대)
       ↓
5. 중간 점검 ← 분석 결과를 보고 모듈 간 의존성 파악, 계획 수정
       ↓
6. 2단계 실행 (병렬 포팅)
       ↓
7. 중간 점검 ← 스텁/placeholder 점검, 통합 에이전트 투입 여부 판단
       ↓
8. 통합 + 검증
       ↓
9. 실행 검증 + 버그 수정 ← 새 세션에서 실행하여 런타임 동작 확인, 버그 수정
```

### 왜 이렇게 해야 하는가

1차 시도에서는 1→4→6을 한 번에 쭉 실행하고, 결과물을 보고 나서야 문제를 발견했다.

- **착수 전 검토**(2단계)로 잡을 수 있는 문제: 통합 에이전트 부재, 인터페이스 명세 누락 등 구조적 결함
- **중간 점검**(5, 7단계)으로 잡을 수 있는 문제: .pas 파일이 에이전트 영역에 걸치는 문제, 검증 깊이 부족, 스텁 발생 등 실행 중에야 드러나는 문제
- **실행 검증**(9단계)으로 잡을 수 있는 문제: 빌드는 성공하지만 런타임에서 발생하는 버그 (리소스 로딩 실패, 로직 오류, 화면 미표시 등)

한 번의 사전 검토로 완벽하게 만들기보다, 작업 흐름 중간에 검토 지점을 두는 것이 현실적이다.

### 실행 검증 단계를 별도로 정의할 것

통합 단계(8단계)까지 완료해도, 실제로 실행해서 동작을 확인하는 것은 별개의 작업이다.
- 빌드 성공 ≠ 정상 동작. 통합 후 실행하면 런타임 버그가 발생할 수 있다
- 통합까지 진행한 세션은 이미 컨텍스트가 크므로, **새 세션에서 실행 검증을 시작**하는 것이 효과적이다
- 같은 세션에서 계속 버그를 잡으면 컨텍스트 압축이 반복되면서 품질이 떨어진다
- CLAUDE.md에 "실행하여 확인한다"는 기준 자체가 없으면, 에이전트는 빌드 성공 시점에서 완료로 간주한다

## 주의사항

- Agent Teams는 아직 Research Preview 단계
- 팀메이트가 작업 완료를 마킹 못하는 경우가 있어 수동 개입 필요
- 팀메이트는 자체 팀을 생성할 수 없음
- 토큰 사용량이 단일 세션 대비 증가 (에이전트 수에 비례)
- `/resume`, `/rewind`로 in-process 팀메이트 복원 불가
