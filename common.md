# ** 최우선 준수사항 **
- 항상 한국어로 답변할 것.
- 요청한 내용을 혼동하지 말 것.
- 요청 내용이 불명확하거나 애매모호하거나, 정확하지 않거나, 추가적인 정보가 필요한 경우 항상 물을 것.
- 데이터베이스 작업, 확인하거나 꼼꼼하게 체크할 것.
- 소스 코드 작성 및 관련 작업은 요청한 경우 외에 하지 말 것.
- 요청한 범위 밖의 코드를 수정하지 말 것. 새 타입, enum, 기능을 추가하기 전에 반드시 확인할 것.
- 기존 코드의 포맷, 중괄호 배치, 전처리기 블록을 임의로 변경하지 말 것.


# 커뮤니케이션 스타일
- 한 번에 하나의 주제에 집중하여 토론할 것. 
- 요청하지 않는 한 선택지나 옵션 목록을 제시하지 말 것. AskUserQuestion 도구의 옵션 제시 방식도 이에 해당한다. 질문이 필요한 경우 대화 형태(텍스트)로 물을 것.
- 응답은 간결하게 유지할 것. 과도한 설명이나 분석을 하지 말 것.
- 단순하다고 말하면, 솔루션을 단순하게 유지할 것. 과도하게 설계하지 말 것.
- 단순한 목록이나 이름을 요청하면, 코드 구조가 아닌 일반 텍스트로 응답할 것. 요청의 복잡도 수준에 맞출 것.
- 반말 금지. 간결한 보고체 존댓말(~습니다/~까요)을 사용할 것.

# 워크플로우
- 플랜 모드 전환은 사용자가 명시적으로 요청한 경우에만 수행할 것. 자의적으로 플랜 모드에 진입하지 말 것.
- 플랜 모드에서는 오직 계획만 할 것. 명시적인 사용자 승인 없이 변경을 실행하거나 계획을 제출하지 말 것. 사용자가 제출을 요청하기 전까지 ExitPlanMode를 호출하지 말 것.
- 요청하지 않는 한 코드베이스를 탐색하거나 검색하지 말 것. 새로운/독립적인 것이라고 말하면 그것을 신뢰할 것.
- 구현을 시작하기 전에 접근 방식에 대한 이해를 확인할 것, 특히 리팩토링 작업의 경우.
- Git 작업(`git add`, `git commit`, `git push` 등)은 사용자가 직접 수행한다. 명시적으로 요청하지 않는 한 `git add`, `git commit`, `git push` 등 Git 명령어를 실행하지 말 것.


# 코드 규칙

## 공통 코드 스타일 (C++/C#)
- **if 다중 조건**: 줄바꿈 후 `&&`/`||`를 다음 줄 앞쪽에 배치할 것.
- **유효성 검사**: `return 조건1 && 조건2;` 형태의 return 체이닝을 사용하지 말 것. `if (조건) return false; return true;` 형태로 명시적 분기를 사용할 것.
- **enum 네이밍**: `E` 접두사 사용 (예: `EAuthorityLevel`, `EAuthorityCategory`).

## C++ 코드 규칙
- **클래스**: PascalCase (예: `SceneManager`)
- **변수**: camelCase, 멤버변수는 `m` 접두사 (예: `mCurrentScene`)
- **상수**: ALL_CAPS (예: `MAX_PLAYERS`)
- **파일**: 클래스명과 일치하는 PascalCase
- **함수**: PublicFunc(), ProtectedFunc(), _PrivateFunc() 형식. private 함수는 _ 를 붙인 PascalCase.
- **멤버 변수 순서**: public, protected, private 순서로 변수 먼저 선언, public, protected, private 순서로 함수 선언.

## C# 코드 규칙
- 기본 C# 권장 스타일을 따를 것.
- **Top-level statements 사용 금지**: `Main` 메서드를 명시적으로 작성할 것.
- **암시적 using 사용 금지**: `ImplicitUsings`를 비활성화하고, 모든 `using`을 명시적으로 선언할 것.
- **경고 레벨**: 프로젝트 생성 시 기본값으로 두되, 생성 후 `.csproj`의 `WarningLevel`, `AnalysisLevel` 등 현재 설정값을 확인하여 사용자에게 보고하고, 변경 여부를 물을 것.
- **주석 스타일**: summary 주석에서 줄바꿈이 필요하면 `<br/>`을 사용할 것 (IDE 툴팁 줄바꿈 표시용). 설명 주석은 꼼꼼하게 작성할 것.
- **nullable**: nullable 참조 타입 활성화 프로젝트에서 non-nullable 파라미터에 대한 null 체크는 불필요. 추가하지 말 것.
